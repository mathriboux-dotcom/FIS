📁 Playbooks

📌 À quoi servent les playbooks ?

Les playbooks pilotent les différentes étapes du déploiement. Ils permettent d’organiser les actions à réaliser dans un ordre précis afin de garantir une installation cohérente sur tous les serveurs. Durant mon stage, j’ai développé le playbook principal utilisé pour le déploiement de FIS Relay.

🔍 Logique générale

  1. Vérification de l’environnement : Contrôle des paramètres, des fichiers et des configurations nécessaires au déploiement.
  2. Préparation du serveur : Création des dossiers et préparation des éléments nécessaires à l’installation.
  3. Sauvegarde de l’installation existante : Conservation de la version précédente afin de pouvoir revenir en arrière en cas de problème.
  4. Installation de la nouvelle version : Déploiement et installation automatique de l’application.
  5. Déploiement des configurations : Mise en place des fichiers de configuration nécessaires au fonctionnement de FIS Relay.
  6. Vérification : Contrôle du bon déroulement du déploiement.
  7. Démarrage du service : Lancement de l’application et vérification de son bon fonctionnement.

💻 Exemple réel

📁 Inventories

📌 À quoi servent les inventories ?

Les inventories regroupent les informations permettant d’identifier les serveurs utilisés lors des déploiements. Ils indiquent notamment quelles machines sont concernées par une opération et permettent aux playbooks de savoir où exécuter les différentes actions prévues. Dans un environnement comportant plusieurs serveurs, ils facilitent l’organisation et la gestion des déploiements.

🔍 Rôle dans le déploiement

Les inventories constituent un lien entre les serveurs et les playbooks. Ils permettent :

  • D’identifier les machines concernées ;
  • De regrouper les serveurs par environnement ;
  • D’associer les paramètres nécessaires au déploiement ;
  • D’assurer l’exécution des opérations sur les bonnes machines.

Grâce à cette organisation, un même playbook peut être utilisé sur différents environnements tout en conservant des configurations adaptées.

📁 FIS Relay

Le rôle FIS Relay est l’un des principaux projets sur lesquels j’ai travaillé durant mon stage. Lorsque je suis arrivée sur ce projet, l’application existait déjà et fonctionnait sur de nombreux serveurs. Chaque serveur possédait sa propre configuration.

Le problème : Beaucoup d’informations étaient identiques d’un serveur à l’autre. Les mêmes paramètres apparaissaient parfois dans plusieurs dizaines de fichiers. Lorsqu’une modification était nécessaire, il fallait vérifier et corriger chaque fichier individuellement. Cette méthode fonctionnait, mais elle demandait beaucoup de temps et augmentait le risque d’erreur.

Mon travail a donc consisté à rendre cette organisation plus simple, plus claire et plus facile à maintenir.

📄 Les commons

La première étape consistait à identifier les informations communes entre les serveurs. Prenons un exemple simple : Imaginons dix serveurs utilisant exactement le même paramètre. Sans organisation particulière, cette information doit être écrite dix fois. Si ce paramètre change, il faut alors modifier dix fichiers différents.

C’est exactement ce type de problème que les commons permettent de résoudre. Les commons regroupent les informations qui doivent être identiques partout. Ils servent de base commune au projet.

Pour préparer ces commons, j’utilisais plusieurs scripts capables d’analyser automatiquement les configurations existantes :

  • build_all_common_ini_prod.sh : analysait les configurations de production.
  • build_all_common_ini_uat.sh : réalisait le même travail pour les environnements de recette.

Les résultats étaient ensuite utilisés pour construire les commons globaux que je créais manuellement. Grâce à cette organisation, une modification importante peut être réalisée une seule fois puis réutilisée partout.

⚙️ Les fichiers .sub

Une fois les informations communes séparées, il restait à gérer les différences entre les serveurs. Certaines informations doivent rester propres à chaque machine, comme :

  • Un identifiant ;
  • Un nom de serveur ;
  • Une adresse spécifique ;
  • Un paramètre technique particulier.

Ces informations ne peuvent pas être placées dans les commons. Elles sont donc stockées dans des fichiers appelés .sub. On peut comparer un fichier .sub à une carte d’identité propre à une machine :

  • Le common contient ce qui est partagé.
  • Le .sub contient ce qui est unique.

Pour éviter de créer ces fichiers manuellement, j’ai développé plusieurs scripts capables de les générer automatiquement. Le script build_all_sub_fisrelay.sh est l’un des principaux scripts utilisés pour cette étape. Il compare les configurations existantes avec les commons et conserve uniquement les valeurs spécifiques à chaque serveur.

Cette automatisation permet de gagner du temps et réduit fortement les risques d’erreur.

🌍 Les environnements

L’application n’est pas utilisée directement en production. Chaque modification passe d’abord par plusieurs étapes de validation. Pour cette raison, l’organisation du rôle FIS Relay tient compte de plusieurs environnements :

  • Production ;
  • Staging (recette) ;
  • Common (paramètres partagés).

Les informations communes sont partagées. Les différences sont isolées. Les paramètres spécifiques sont générés automatiquement. Cette structure permet d’obtenir des configurations cohérentes tout en conservant les particularités de chaque environnement.

📋 Intégration dans Ansible

Une fois les fichiers préparés, ils sont intégrés dans Ansible. Ansible est l’outil chargé d’automatiser les déploiements.

Il récupère :

  1. Les commons ;
  2. Les fichiers .sub ;
  3. Puis il reconstruit automatiquement la configuration finale qui sera utilisée sur le serveur.

Cette étape permet d’éviter les modifications manuelles et garantit que chaque machine reçoit une configuration cohérente. Le fichier main.yml coordonne ensuite l’ensemble des opérations. Il joue le rôle de chef d’orchestre du déploiement.

---
- name: Apply FIS Relay configuration

  become: yes

  tasks:

📁 Scripts build_common

📝 Introduction

Lorsque plusieurs serveurs utilisent une même application, certaines informations se retrouvent dans de nombreuses configurations. Avant de pouvoir créer des fichiers communs réutilisables, il est nécessaire d’analyser ces configurations et de vérifier qu’elles respectent les règles du projet. Les scripts de ce dossier automatisent l’ensemble de ce processus.

⚙️ Ce que font ces scripts

Ces scripts lancent automatiquement l’analyse des environnements de production et de recette. Ils exécutent ensuite le checker, un outil de contrôle chargé de vérifier la cohérence des fichiers générés. Une fois les vérifications terminées, ils préparent les résultats qui seront utilisés pour construire les commons. Ils jouent donc le rôle de chef d’orchestre du processus.

🚀 Ce que j’ai réalisé

J’ai travaillé sur l’amélioration et la maintenance de ces scripts afin d’automatiser l’ensemble de la génération des commons. J’ai également participé à l’intégration du checker et à l’organisation des différentes étapes du traitement. Cette automatisation permet de réduire les tâches manuelles et de détecter plus rapidement les erreurs de configuration.

📄 run_all_common.sh

Ce script lance automatiquement l’analyse des environnements de production et de recette. Il exécute ensuite le checker, un outil de contrôle chargé de vérifier la cohérence des fichiers générés.

📄 cleaned_ini.sh

Après le passage du checker, plusieurs fichiers intermédiaires sont générés. Ces fichiers doivent être nettoyés avant d’être réutilisés. Le script cleaned_ini.sh intervient à cette étape. Il récupère les fichiers validés par le checker, supprime les éléments inutiles et prépare une version plus propre des configurations.

📄 generate_p3_server_common_csv.sh

Avant de créer un common, il est important de comprendre quelles informations sont réellement identiques entre les différents serveurs. Comparer plusieurs dizaines de fichiers à la main serait long et peu fiable. J’ai donc utilisé ce script pour automatiser cette analyse et obtenir une vue globale des configurations existantes.

Ces fichiers permettent de visualiser rapidement quelles valeurs sont communes et quelles valeurs diffèrent selon les serveurs.

📄 build_csv_compare_all_ini.sh

Lorsque plusieurs serveurs utilisent une même application, leurs configurations sont souvent très proches. Cependant, quelques différences peuvent exister d’une machine à l’autre. Avant de créer un common, il est donc nécessaire de savoir précisément quels paramètres sont identiques et lesquels doivent rester spécifiques.

Ce script parcourt automatiquement les différents fichiers INI présents dans les environnements. Il extrait les informations contenues dans ces fichiers puis les compare entre tous les serveurs. Il génère ensuite des rapports CSV qui indiquent :

  • Les paramètres identiques ;
  • Les paramètres différents ;
  • Les valeurs manquantes ;
  • Les incohérences éventuelles.

📄 build_all_common_ini_prod.sh

Lorsque plusieurs serveurs de production utilisent la même application, une grande partie de leurs paramètres est souvent identique. Conserver ces informations dans chaque fichier rend les configurations plus difficiles à maintenir. Ce script a été créé pour identifier automatiquement ces éléments communs et préparer la création des commons utilisés par Ansible.

Le script analyse les fichiers de configuration provenant des serveurs de production. Il compare les différentes valeurs afin de repérer celles qui sont présentes partout. Lorsqu’un paramètre est identique sur plusieurs machines, il peut être déplacé dans un common.

Cet environnement permet de vérifier que les configurations fonctionnent correctement avant leur utilisation réelle. Comme pour la production, certaines informations sont communes à plusieurs serveurs et peuvent être regroupées.

📁 Scripts build_sub_ini_ansible

📝 Introduction

Une fois les informations communes regroupées dans les commons, il reste encore les paramètres propres à chaque serveur. Ces informations ne peuvent pas être partagées car elles dépendent directement de la machine concernée. C’est pour cette raison que le projet utilise des fichiers .sub.

⚙️ Ce que font ces scripts

Ces scripts analysent les configurations déjà présentes sur les serveurs. Ils comparent ces fichiers avec les commons précédemment créés. Les informations déjà présentes dans les commons sont ignorées. Seules les valeurs spécifiques à chaque serveur sont conservées. Ces informations sont ensuite enregistrées dans les fichiers .sub.

🚀 Ce que j’ai réalisé

J’ai travaillé sur l’automatisation de la génération des fichiers .sub. L’objectif était d’éviter la création manuelle de ces fichiers et de réduire les risques d’erreur lors des déploiements.

📄 build_all_sub_fisrelay.sh

Le script analyse les configurations déjà présentes sur les serveurs. Il compare ces fichiers avec les commons précédemment créés. Les informations déjà présentes dans les commons sont ignorées. Seules les valeurs spécifiques à chaque serveur sont conservées. Ces informations sont ensuite enregistrées dans les fichiers fisrelay.sub.

📄 restore_ini_fisrelay.sh

📝 Introduction

Au cours du projet, les configurations étaient volontairement séparées en plusieurs fichiers. Certaines informations étaient regroupées dans les commons, qui contiennent les paramètres partagés entre plusieurs serveurs. D’autres informations étaient stockées dans des fichiers .sub, qui contiennent uniquement les paramètres propres à une machine.

Cette organisation facilite la maintenance des configurations, mais rend leur lecture plus complexe. Pour obtenir la configuration finale réellement utilisée par un serveur, il est nécessaire de reconstituer toutes ces informations. C’est précisément le rôle du script restore_ini_fisrelay.sh.

⚙️ Ce que fait ce script

Ce script reconstruit automatiquement un fichier de configuration complet à partir des différents fichiers utilisés par Ansible. Pour cela, il commence par identifier le serveur concerné ainsi que l’environnement associé. Il récupère ensuite les différentes couches de configuration utilisées par le projet :

  • Les commons globaux ;
  • Les commons de l’environnement ;
  • Les fichiers spécifiques au serveur ;
  • Les fichiers .sub générés automatiquement.

Ces différentes sources sont ensuite fusionnées dans le bon ordre afin de produire un fichier unique appelé fisrelay_generate.ini. Ce fichier représente la configuration finale qui sera réellement utilisée après le déploiement. Le script prend également en compte certaines configurations complémentaires comme celles du rôle Load Balancing lorsqu’elles existent.

🔎 Vérification avec le checker

Une fois la configuration reconstruite, le script peut également comparer le résultat obtenu avec les configurations présentes sur les serveurs. Pour cela, il utilise le checker. Le checker est un outil de contrôle permettant d’analyser un fichier de configuration et de vérifier qu’il respecte les règles attendues par l’application.

Il permet notamment de détecter :

  • Des paramètres manquants ;
  • Des différences inattendues ;
  • Des incohérences dans la configuration ;
  • Des valeurs supprimées ou modifiées.

📊 Génération du rapport CSV

En plus du fichier INI final, le script peut produire automatiquement un rapport CSV. Ce rapport compare :

  • Les commons ;
  • Les fichiers de configuration d’origine ;
  • Les résultats du checker ;
  • Le fichier généré par le script.

Chaque ligne reçoit ensuite un statut permettant d’identifier rapidement le résultat de la comparaison, comme :

  • OK
  • OK_RESOLU_SUB
  • MANQUE_ORIGIN
  • DIFFERENT
  • DIFFERENT_CHECKER

🚀 Ce que j’ai réalisé

J’ai travaillé sur l’amélioration et la maintenance de ce script afin de faciliter les vérifications de configuration. J’ai notamment participé à :

  • L’intégration des différentes couches de configuration utilisées par Ansible ;
  • La prise en compte des fichiers .sub ;
  • L’exploitation des résultats produits par le checker.

J’ai également utilisé les rapports CSV générés par le script pour comparer les configurations et identifier plus rapidement d’éventuelles incohérences. Grâce à cet outil, il devient possible de visualiser facilement la configuration finale d’un serveur sans avoir à reconstituer manuellement l’ensemble des fichiers utilisés par le projet.

🚀 Jenkins

📝 Introduction

Une fois les fichiers de configuration préparés, il faut encore les déployer sur les serveurs. Pour éviter d’effectuer toutes les opérations manuellement, l’entreprise utilise Jenkins. Jenkins est un outil d’automatisation qui permet de lancer, suivre et contrôler les déploiements. Il constitue le point central entre les configurations préparées dans Ansible et les serveurs sur lesquels l’application est installée.

Durant mon stage, j’ai régulièrement utilisé Jenkins afin de lancer des déploiements, vérifier leur déroulement et contrôler que les configurations générées étaient correctement appliquées.

⚙️ Préparer un déploiement

Avant de lancer un déploiement, plusieurs informations doivent être renseignées, comme :

  • L’environnement cible ;
  • Le serveur concerné ;
  • La version de FIS Relay à installer ;
  • Certaines options de vérification ou de simulation.

Cette étape permet de contrôler précisément ce qui sera déployé et où les modifications seront appliquées.

🔗 Jenkins dans le processus global

Jenkins n’agit pas seul. Avant d’arriver dans Jenkins, plusieurs étapes ont déjà été réalisées :

  • Les scripts présentés dans les sections build_common et build_sub_ini_ansible permettent de générer les commons et les fichiers .sub utilisés par Ansible.
  • Ces fichiers sont ensuite intégrés dans les différents rôles présentés dans les rubriques : FIS Relay, Calendar, EnrichData, Load Balancing, Session Chooser.

Une fois ces configurations prêtes, Jenkins prend le relais afin de lancer automatiquement le déploiement. Le détail du fonctionnement des playbooks est présenté dans la rubrique Playbooks du site. Ils permettent ensuite d’appeler les différents rôles Ansible ainsi que leurs fichiers main.yml.

📄 Le rôle du Jenkinsfile

Derrière l’interface Jenkins se trouve un fichier appelé Jenkinsfile. Ce fichier contient toute la logique du pipeline de déploiement. On peut le comparer à une recette qui décrit chaque étape à exécuter.

Le Jenkinsfile permet notamment de :

  • Récupérer les paramètres choisis par l’utilisateur ;
  • Préparer l’environnement ;
  • Lancer les playbooks Ansible ;
  • Suivre l’exécution du déploiement ;
  • Effectuer les vérifications finales.

Dans le cadre de mon stage, le fichier Jenkinsfile_fisrelay était l’un des éléments centraux du processus de déploiement. J’ai étudié son fonctionnement afin de comprendre comment les paramètres renseignés dans Jenkins étaient transmis à Ansible et utilisés lors de l’installation de l’application.

🚀 Ce que j’ai réalisé

J’ai pris le temps d’analyser le fonctionnement du Jenkinsfile utilisé pour FIS Relay afin de comprendre l’ensemble de la chaîne de déploiement. Cela m’a permis de suivre le parcours complet d’une configuration : depuis sa génération par les scripts jusqu’à son installation automatique sur les serveurs.
